# Commit

## Goals

- Commit only the intended Aident changes.
- Follow the repo's required commit prefixes.
- Avoid history edits that would surprise reviewers.

## Steps

1. Read `AGENTS.md` and `CLAUDE.md` for repo conventions before committing.
2. Inspect the working tree with `git status`, `git diff`, and `git diff --staged`.
3. Stage only the intended files with `git add`.
4. If you are still on `main` or a detached HEAD, stop and create a feature
   branch first with `gt branch create <branch-name>`.
5. Choose a short one-line message with one of the required prefixes:
   `[fix]`, `[feat]`, `[refactor]`, `[docs]`, `[test]`, `[chore]`.
6. Prefer `gt commit create -m "<message>"` for a new commit.
7. Use `gt commit amend -m "<message>"` only when intentionally updating the
   most recent commit from the current run. Do not amend someone else's commit
   history unless the task explicitly requires it.
8. Reply with the final commit SHA and message.

## Notes

- Never add `Co-authored-by`, `Generated by`, or any other AI attribution.
- Keep commit messages concise; Aident does not require a multi-line body by
  default.
